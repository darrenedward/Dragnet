### Security Rules

Aligned with OWASP Top 10. Non-negotiable.

- [SECURITY] Always use HTTPS in production: Authentication cookies and tokens must be transmitted over HTTPS. Set secure flag on cookies. Test auth flows with secure flags enabled. _(Better Auth)_
- [SECURITY] Server-side validation is required: Always validate and sanitize user input on the server. Client-side validation can be bypassed. Server Actions must validate inputs before processing. _(Next.js)_
- [SECURITY] Raw SQL queries are dangerous: Avoid prisma.$queryRaw unless absolutely necessary. Always use parameterized queries with Prisma.sql tagged template to prevent SQL injection. _(Prisma)_
- [CORRECTNESS] Upgrade better-auth to fix 7 vulnerabilities including auth bypass and open redirect: Upgrade better-auth from v1.2.8 to v1.6.14 or later. Versions below 1.6.2 contain 7 security flaws.

**Unauthenticated API key creation** (GHSA-99h5, fixed 1.3.26): api-key plugin lets unauthenticated users create keys. Upgrade to >=1.3.26.

**Rate limiter IPv6 bypass** (GHSA-p6v2, fixed 1.4.17): rotating IPv6 prefixes circumvent rate limiting. Upgrade to >=1.4.17.

**OAuth state mismatch** (GHSA-wxw3, fixed 1.6.2): OAuth callbacks accept mismatched state parameters without PKCE verification. Upgrade to >=1.6.2.

### Component Best Practices

Component-specific patterns that have been validated at scale.

- [CORRECTNESS] Implement proper session management: Set appropriate session expiration times and implement refresh token rotation. Use the session plugin for device management and forced logout. _(Better Auth)_
- [CORRECTNESS] Avoid 'any' type: Use specific types instead of 'any'. If you don't know the type, use 'unknown' and add type guards. Enable noUncheckedIndexedAccess and strict mode. _(TypeScript)_
- [CORRECTNESS] Use transactions for multi-step operations: Wrap complex operations in prisma.$transaction() to ensure atomicity. Use interactive transactions for conditional logic within the transaction. _(Prisma)_
- [CORRECTNESS] Use Server Components as the default: React Server Components reduce client bundle size by keeping rendering on the server. Use them for data fetching, accessing backend resources, and static content rendering. _(React)_
- [PERFORMANCE] Use Server Actions for data mutations: Prefer Server Actions over API routes for form submissions and data mutations. They provide better type safety, reduce client-side code, and integrate with progressive enhancement. _(Next.js)_
- [PERFORMANCE] Use satisfies for type validation: The 'satisfies' operator validates that an expression matches a type without widening or losing literal types. Prefer it over type annotations for constant objects. _(TypeScript)_
- [PERFORMANCE] Use @apply sparingly: Tailwind v4 has native cascade layers and zero-config. Prefer component extraction over @apply. Use CSS variables for theming instead of config files. _(Tailwind CSS)_
- [PERFORMANCE] Keep components small and focused: Each component should have a single responsibility. If a component handles too many concerns, break it into smaller components with clear props interfaces. _(React)_
- [PERFORMANCE] Use workspace config for monorepos: Vitest workspaces allow different test configurations per package. This enables isolated testing with shared caching in monorepo setups. _(Vitest)_

### Performance Rules

Aligned with Lighthouse scoring. Measure before and after.

- [CORRECTNESS] Leverage Static Generation where possible: Use generateStaticParams for dynamic routes to pre-render pages at build time. Combine with ISR (Incremental Static Regeneration) for pages that need periodic updates without full rebuilds. _(Next.js)_
- [CORRECTNESS] Content configuration must be accurate: Ensure your content paths in CSS or config cover all files that use Tailwind classes. Missed paths mean purged classes. Tailwind v4 auto-detects content sources. _(Tailwind CSS)_
- [PERFORMANCE] Use project references for monorepos: Project references enable incremental compilation across packages. This dramatically speeds up builds in monorepo setups. _(TypeScript)_
- [PERFORMANCE] Use select to limit fields returned: Always use select when you don't need all fields. This reduces memory usage and network transfer. Prisma's type system will infer the correct return type. _(Prisma)_
- [PERFORMANCE] Enable browser mode for DOM tests: Use browser mode with Playwright for tests that depend on real DOM APIs. It is more reliable than jsdom and catches browser-specific issues. _(Vitest)_
- [CONVENIENCE] Use React.memo judiciously: Only use React.memo when profiling shows a measurable benefit. Over-memoization adds complexity without payoff. In React 19, compilers can handle most memoization automatically. _(React)_

### Constraints

Hard boundaries. Do not cross without explicit approval.

- [CORRECTNESS] Avoid N+1 queries with includes: When fetching related data in loops, use include or fluent API instead of individual queries. For complex joins, consider raw queries or database views. _(Prisma)_
- [CORRECTNESS] Don't test implementation details: Test behavior, not implementation. Avoid testing internal state or private methods. Tests coupled to implementation break on refactors and provide false confidence. _(Vitest)_
- [PERFORMANCE] Avoid client components for static content: Default to Server Components. Only add 'use client' when you need interactivity, browser APIs, or React hooks. Unnecessary client components increase bundle size. _(Next.js)_
- [PERFORMANCE] Don't use type assertions excessively: Type assertions (as Type) bypass the type checker. Use type guards, discriminated unions, or generics instead. Every 'as' is a potential runtime error. _(TypeScript)_
- [PERFORMANCE] Avoid prop drilling beyond 2 levels: When props pass through more than 2 intermediary components, use context, a state manager, or component composition instead. _(React)_
- [CONVENIENCE] Avoid inline arbitrary values for repeated patterns: If the same arbitrary value (e.g., text-[#1da1f2]) is used in multiple places, define it as a design token or theme extension instead. _(Tailwind CSS)_

## CRITICAL: Use ripgrep, not grep

NEVER use grep for project-wide searches (slow, ignores .gitignore). ALWAYS use rg.

- `rg "pattern"` — search content
- `rg --files | rg "name"` — find files
- `rg -t python "def"` — language filters

## File finding

- Prefer `fd` (or `fdfind` on Debian/Ubuntu). Respects .gitignore.

## JSON

- Use `jq` for parsing and transformations.

- Cap reads at 250 lines; prefer `rg -n -A 3 -B 3` for context
- Use `jq` for JSON instead of regex
- Existing lint baseline includes long-standing `react/no-array-index-key` warnings (example: `src/components/treaties/ApplicantInformationSection.tsx:137:21`); do not treat these as regressions unless explicitly asked to fix them.

# Treaty System Architecture

## Overview

The NWA Alliance system implements a three-level hierarchical treaty structure that governs user relationships with treaties and businesses:

1. **Treaties** (Level 1): High-level treaty agreements (e.g., "Peace & Trade Treaty", "Farmers Treaty")
2. **Treaty Types** (Level 2): Business categories under each treaty (e.g., "Small Farm", "Large Church", "Micro Home Business")
3. **User Relationships** (Level 3): User-treaty and user-treaty-type assignments

## Database Schema Relationships

```sql
-- Core Treaty Tables
treaties (id, name, description, is_active)
treaty_types (id, name, category, description, is_active, cost)

-- Bridge Table - Treaty to Treaty Type Relationships
treaty_treaty_types (treaty_id, treaty_type_id)

-- User Assignment Tables
user_treaties (user_id, treaty_id, status, assigned_date)
user_treaty_numbers (user_id, treaty_type_id, treaty_number, is_active)
```

## User Assignment Workflow

### Prerequisites for Treaty Type Assignment
1. **User MUST have a Treaty** - Users cannot have treaty types without first having a treaty
2. **User MUST have a Treaty Number** - Validated treaty number format: 3 letters + 6 alphanumeric characters
3. **Treaty Types are Filtered** - Users can only select treaty types associated with their assigned treaty

### Assignment Process
1. **Treaty Selection**: User selects from available treaties
2. **Treaty Type Filtering**: System filters treaty types based on selected treaty
3. **Business Details Collection**: Logo, name, address, legal documents, business categories
4. **Cost Assignment**: Treaty type cost is added to user account
5. **Payment Tracking**: Manual payment process managed by admins

## Implementation Requirements

### Frontend (React/Next.js)
- Treaty dropdown selection in UserIdentificationForm
- Filtered treaty type dropdown based on selected treaty
- Business details form with file upload capability
- Payment status display and management interface

### Backend (API Routes)
- Treaty validation and filtering logic
- Treaty number format validation (regex: `^[A-Za-z]{3}[A-Za-z0-9]{6}$`)
- User-treaty-treaty-type relationship management
- Unique constraint handling with skipDuplicates

### Display Logic
- Show treaty name selected by user (NOT treaty type name)
- Pre-populate all user data in UPDATE forms
- Refresh data after save operations to ensure persistence

## Business Logic Rules

1. **Hierarchical Structure**: Treaty → Treaty Types → User Assignments
2. **Unique Constraints**: (user_id, treaty_type_id) combinations must be unique
3. **Cost Management**: Each treaty type has associated cost for manual payment tracking
4. **Permission System**: Treaty assignment follows RBAC system with proper permission checks

## Data Validation

### Treaty Number Format
- Pattern: 3 letters + 6 alphanumeric characters
- Example: "PPT435098", "FARM123456"
- Validation: Backend regex with error handling

### Database Constraints
- User cannot have duplicate treaty type assignments
- Treaty types must be active to be assigned
- Treaty numbers must be unique across users

## Agent Implementation Guidelines

When working with treaty system features:

1. **Always check treaty hierarchy** - Users need treaty before treaty types
2. **Validate treaty numbers** - Use backend validation with proper error messages
3. **Filter treaty types** - Based on user's assigned treaty
4. **Display correct names** - Show treaty name, not treaty type name in treaty display
5. **Handle data persistence** - Refresh user data after save operations
6. **Follow unique constraints** - Use skipDuplicates for database operations
