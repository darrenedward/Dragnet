## 1. Summary

**GrepLoop** is a local-first, Greptile-style automated code review tool that runs entirely on the developer's machine. It watches one or more registered local git repositories for new branches (treated as "PRs" in the absence of a remote host), diffs them against a base branch, and runs an AI-powered review pass — using either a cloud LLM API or a fully local model — producing a structured review report.

It supports two trigger modes per project:
1. **Auto-inspect** — any new branch matching a configured pattern is reviewed automatically once it stabilizes (no new commits for N minutes, configurable).
2. **Mention-triggered** — review only runs when the user adds a trigger marker (e.g. a `@PRBot` line in a commit message, a local `.greploop-review` marker file, or a CLI command) — mirroring how Greptile/CodeRabbit let you choose "review everything" vs. "review on request."

No GitHub/GitLab account, token, or network access is required for the MVP. All output is written to a local report (Markdown + a local web viewer), never posted anywhere.

---

## 2. Problem Statement

Tools like Greptile and CodeRabbit provide automated, context-aware PR review — but they require pushing code to a remote host and granting a third-party service access to the repository. For local-only projects, experimentation branches, client work under NDA, air-gapped environments, or developers who simply don't want their code touching a SaaS review tool, there's no equivalent. Developers either review their own diffs manually or skip automated review entirely for local work.

GrepLoop closes that gap: the same category of review (correctness, security, style, missed edge cases) but running entirely offline against local git state.

---

## 3. Goals & Non-Goals

### Goals (MVP)
- Watch N local repos for new "PR-like" branches with zero manual setup beyond initial registration.
- Diff a candidate branch against a configurable base branch using plain git, no remote required.
- Support both auto-inspect and mention-triggered review modes, per-project.
- Run review via a pluggable LLM backend: cloud API (Claude/OpenAI-compatible) or local model (Ollama or compatible), configurable per project or globally.
- Produce a structured, categorized review report (see Section 6) as Markdown, viewable in a local lightweight web UI.
- Be genuinely useful for solo use without requiring any account, server, or cloud dependency.

### Non-Goals (MVP — explicitly deferred)
- Posting comments back to GitHub/GitLab/Bitbucket PRs.
- Multi-user authentication, teams, or org-level permissions.
- Real remote-PR polling via host APIs.
- IDE-embedded inline annotations (VS Code extension) — a Phase 2 candidate, not MVP.
- Review of non-text/binary diffs.

### Explicit "shaped for later, not built now"
The MVP is single-user, but a few cheap architectural choices keep multi-user/remote expansion realistic later without over-engineering now (see Section 8). This is a deliberate middle ground: not full multi-tenancy, but not a structure that has to be torn out either.

---

## 4. Users & Use Cases

**Primary user (MVP):** A solo developer running GrepLoop on their own machine against their own local repos.

**Core use cases:**
1. *"I opened a feature branch and want an automatic sanity check before I merge to main, without lifting a finger."* → Auto-inspect mode.
2. *"I'm mid-flow on a branch and don't want review noise on every commit — I'll ask for it when I'm ready."* → Mention-triggered mode.
3. *"I want to register a new repo for watching in under a minute."*
4. *"I want to see review history for a branch, not just the latest pass."*
5. *(Stretch, explicitly named by user as a "maybe later")* "I want to point this at a colleague's repo or run it as a small shared service for my team."

## 5. Core Concepts & Terminology

Since there's no remote host, GrepLoop redefines a few familiar terms for the offline context:

| Term (as used in this PRD) | Meaning |
|---|---|
| **Local PR** | A git branch in a watched repo that differs from its configured base branch and matches the "PR-like" criteria (see 5.1). Not a GitHub/GitLab PR object — there is no remote. |
| **Base branch** | The branch a local PR is diffed against (typically `main` or `develop`), configurable per repo. |
| **Watched project** | A local repo path registered with GrepLoop for monitoring. |
| **Review pass** | One execution of the LLM-backed review pipeline against a specific diff snapshot. |
| **Trigger mode** | Per-project setting: `auto` or `mention`. |

### 5.1 What counts as a "Local PR"?
A branch is considered review-eligible when:
- It is not the configured base branch itself.
- It has at least one commit not present on the base branch.
- It optionally matches a configured branch name pattern (e.g. `feature/*`, `fix/*`) — if no pattern is set, all non-base branches qualify.

This is intentionally simple for MVP: GrepLoop is reading `.git` state directly via local git commands (`git for-each-ref`, `git diff`, `git log`), not parsing any PR-specific metadata, because none exists offline.

## 6. Trigger Modes (Detailed)

### 6.1 Auto-inspect mode
- A filesystem/git watcher polls (or uses OS-level file watching on `.git/refs/heads` and `.git/logs/HEAD`) for branch changes.
- When a qualifying branch is detected, GrepLoop waits for a configurable **quiet period** (default: 5 minutes of no new commits) before triggering review, to avoid reviewing a branch mid-edit.
- Re-review on subsequent pushes: if new commits land after a review has already run, GrepLoop queues a fresh pass after the next quiet period, and the report shows pass history (not just latest).

### 6.2 Mention-triggered mode
Three trigger mechanisms, all offline, in order of friction (low to high):
1. **CLI command** — `greploop review <branch>` — explicit, zero ambiguity, recommended primary path for MVP.
2. **Marker file** — presence of a `.greploop-review` file (empty, or containing flags) anywhere in the branch's working tree triggers review on next watcher poll, then the marker is consumed/cleared.
3. **Commit message marker** — a commit message containing `@PRBot review` (configurable string) triggers review of that branch. This mirrors the GitHub-comment-mention pattern the user described, adapted to a no-remote context where there are no PR comments to scan — so the commit message is the closest offline equivalent.

The user's original idea (a literal `@PRBot` comment on a PR) doesn't have a direct offline analog since there's no PR comment thread without a remote host — option 3 above is the closest faithful adaptation, and option 1 (CLI) is the most reliable for MVP. The marker-file approach is a middle ground for people who want a "drop a file, get a review" workflow without touching the terminal.

### 6.3 Per-project configuration
Each watched project has its own settings, stored in a local config file (not environment variables, so multiple projects can coexist):

```yaml
# ~/.greploop/projects/my-app.yaml
path: /home/user/code/my-app
base_branch: main
trigger_mode: auto          # auto | mention
quiet_period_minutes: 5
branch_pattern: "feature/*" # optional, omit for "all branches"
llm_backend: local          # local | cloud | inherit (use global default)
review_categories:          # subset of Section 7 categories; omit for "all"
  - correctness
  - security
```

## 7. Review Pipeline & LLM Backend

### 7.1 Pipeline stages
1. **Detect** — watcher identifies a qualifying branch + trigger condition met.
2. **Diff extraction** — `git diff <base>...<branch>` plus full changed-file contents (not just diff hunks) for files under a configurable size threshold, to give the LLM enough context beyond bare diff lines.
3. **Chunking** — large diffs are split by file, with a per-file token budget; files exceeding the budget are diffed-only (no full content) with a note in the report.
4. **Review generation** — diff + file context sent to the configured LLM backend with a structured prompt asking for findings categorized per Section 8.
5. **Report assembly** — findings are parsed, deduplicated, and rendered to Markdown + structured JSON (for the local web viewer).
6. **Storage** — report saved to `~/.greploop/reports/<project>/<branch>/<timestamp>.md` (and `.json`), with an index for history.

### 7.2 LLM backend — configurable, both supported
Per your requirement, both options are first-class, selectable per-project or globally:

| Backend | How it works | Tradeoffs |
|---|---|---|
| **Cloud API** | Claude API / OpenAI-compatible endpoint, key stored in local OS keychain (not plaintext config) | Higher review quality typically, requires network + API key + per-call cost, diff content leaves the machine |
| **Local model** | Ollama (or any OpenAI-compatible local server) running on the same machine | Fully offline, no per-call cost, no data leaves machine, quality depends heavily on model size/choice and local hardware |

GrepLoop itself doesn't bundle or manage model weights — it talks to whichever local server (Ollama, LM Studio, etc.) the user already has running, via a configurable endpoint URL. This keeps GrepLoop's own footprint small and avoids it becoming a model-distribution problem.

A global default is set in `~/.greploop/config.yaml`; any project can override it. This is the "configurable" approach you specified rather than a hard global switch.

## 8. Review Criteria & Metrics (derived from legitimate standards in the source file)

This is the section that pulls from the *trustworthy* generic content in your `claude.md` — these are well-established, widely-used standards (OWASP, WCAG, Core Web Vitals, standard TS/lint conventions), not anything specific to the fabricated stack. They're reframed here as **review categories the bot checks for in the code it's reviewing**, not as rules GrepLoop itself must obey while being built.

Each category below can be toggled on/off per-project (see `review_categories` in 6.3).

### 8.1 Correctness
- Logic errors, off-by-one issues, incorrect conditionals.
- Unhandled error paths / missing error handling.
- Type safety gaps (flagging `any`/untyped escape hatches in typed languages, missing null checks).
- Test coverage gaps for changed logic (flagged, not blocking — GrepLoop reports, it doesn't gate).

### 8.2 Security (OWASP-aligned categories)
- Injection risks: unparameterized queries, string-concatenated SQL, unsanitized input reaching dangerous sinks.
- Broken access control: missing auth/authorization checks on new endpoints or handlers.
- Cryptographic issues: weak hashing for secrets/passwords, plaintext secret storage, missing HTTPS-only flags on cookies.
- Hardcoded secrets/credentials in diffs.
- SSRF-shaped patterns: unsanitized user-supplied URLs used in server-side requests.

These map to OWASP Top 10 categories generically — GrepLoop is not OWASP-certified tooling and findings should be treated as a first pass, not a security audit.

### 8.3 Performance
- Obvious N+1 query patterns in loops.
- Unbounded loops or recursive calls on user-controlled input.
- Large synchronous operations on a request/render path.
- (Web projects only, auto-detected by file types present) Core Web Vitals-relevant patterns: missing image dimensions, unoptimized image tags, render-blocking patterns.

### 8.4 Accessibility (web projects only, auto-detected)
- Missing alt text on images.
- Non-semantic interactive elements (`<div onClick>` instead of `<button>`).
- Missing form labels.
- Heading hierarchy skips.

This category is only activated when the diff touches frontend/markup files — it's skipped entirely for backend-only or CLI-tool repos so it doesn't generate noise on projects where it's meaning-less.

### 8.5 Style & maintainability
- Inconsistent naming conventions within the changed files.
- Overly large functions/files relative to the rest of the codebase's own norms (GrepLoop infers thresholds from the existing codebase rather than hardcoding a fixed line count, since "500 lines max" is a project-specific convention, not a universal one).
- Dead code / unused imports introduced by the diff.

### 8.6 Report structure
Each review report contains, per finding:
- **Category** (one of 8.1–8.5)
- **Severity** (`blocker` / `warning` / `suggestion`) — GrepLoop never auto-blocks a merge; severity is informational only, since there's no merge gate to attach to in a local-only tool.
- **File + line reference**
- **Explanation** in plain language
- **Suggested fix** (if confidently generatable) as a diff snippet, clearly marked as a suggestion, not auto-applied.

Summary metrics at the top of each report:
- Total findings by category and severity.
- Files changed / lines changed.
- Review pass duration and backend used (local/cloud + model name) — useful for the user to judge whether a fast local-model pass missed things a cloud pass would catch.

│                               ▼                        │
│                      ┌─────────────────┐               │
│                      │ Diff Extractor   │              │
│                      └────────┬─────────┘               │
│                               ▼                        │
│                      ┌─────────────────┐               │
│                      │ LLM Router       │◀── local/cloud config
│                      └────────┬─────────┘               │
│                               ▼                        │
│                      ┌─────────────────┐               │
│                      │ Report Assembler │               │
│                      └────────┬─────────┘               │
│                               ▼                        │
│                      ┌─────────────────┐               │
│                      │ Local Storage    │               │
│                      │ (~/.greploop)    │               │
│                      └────────┬─────────┘               │

**Form factor:** A background daemon/service (so watching works without a terminal open) plus a CLI for management commands (`greploop add <path>`, `greploop review <branch>`, `greploop status`) plus a minimal local web UI (bound to `localhost` only) for browsing reports. This combination directly answers your "I'm not sure how this works" question on form factor — the daemon is what makes auto-inspect possible at all; without something always running, nothing could "notice" a new branch.

## 10. Multi-user / Expansion Path (explicitly Phase 2+, not built now)

You raised the "maybe build with expansion in mind" question directly — here's the honest answer: build solo-first, but make three cheap choices now that don't cost extra effort today but avoid a rewrite later:

1. **Project registry keyed by ID, not by single global state** — even with one user, store projects as discrete records (already shown in 6.3) rather than one flat global config. Adding a `user_id` field later is trivial; retrofitting it into a single-blob config is not.
2. **LLM Router as a separate module with a clean interface** — already required anyway (local vs cloud), but keeping it decoupled from the rest of the pipeline means a future "per-user API key" or "per-org default backend" slots in without touching the review pipeline.
3. **Local storage path is already namespaced by project** — `~/.greploop/reports/<project>/...` — so adding a user/org layer above that later (`~/.greploop/<org>/<project>/...` or a real DB) is a path-prefix change, not a schema rewrite.

What is explicitly **not** built now: authentication, sessions, server-side multi-tenancy, RBAC, or anything resembling the auth stack referenced in the (suspect) source file. None of that has a place in a single-user local tool, and bolting it on speculatively would be the over-engineering your instinct was right to question. If GrepLoop ever becomes a shared/hosted service, that's a distinct PRD with its own auth model — not a checkbox to flip on this one.
